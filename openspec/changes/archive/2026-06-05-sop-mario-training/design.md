## 背景

我们已有一个可工作的 DDQN 智能体（`mario_rl/`）和基于 SSH 的远程训练工具 `remote_train.py`，部署在一台共享服务器上（2× RTX 2080 Ti，每卡 22GB）。当前工作流程是手动的：SSH 登录、运行实验、scp 取回结果、手工对比。没有结构化的 SOP，没有可复用的分析工具，也没有系统化地追踪哪个超参数配置带来了哪个通关率。

服务器是学生共享的——长时间运行的作业可能导致 GPU 争用。2080 Ti 有 22GB 显存，因此 batch 大小 ≤ 128 和当前模型规模是安全的，但必须避免内存密集操作（例如 GPU 上的超大回放缓冲区、过于宽层的双 Q 网络）。

**约束条件：**
- 远端 Python 3.10；`gym-super-mario-bros 7.4.0`、`gym 0.21.0`、PyTorch 2.5.1+cu121
- 通过 SSH（paramiko）使用 `screen` 会话进行远程训练
- 本地为 Windows + Python；通过 SCP 同步代码
- 单次实验运行时长必须 ≤ 2 小时（共享服务器公平性原则）

## 目标 / 非目标

**目标：**
1. 定义一个分阶段训练 SOP（基线 → 超参数扫描 → 算法调优 → 最终打磨），有明确的决策门槛
2. 构建可复用的 CLI 工具（`scripts/analyze.py`、`scripts/compare.py`）用于跨实验分析——避免每次会话重写分析代码
3. 以通关率（flag_get）为主要 KPI，平均 x 位置为次要，通关耗时为第三
4. 每个实验必须产生一个自包含的结果目录，包含指标、检查点和自动生成的摘要
5. 所有实验对 RTX 2080 Ti（22GB）安全：batch ≤ 64, replay ≤ 200K, frame stack ≤ 4

**非目标：**
- 新 RL 算法（本次变更不涉及 PPO、A3C、Rainbow 等）
- 修改 `gym-super-mario-bros` 环境或奖励函数
- 多 GPU 或分布式训练
- 实时监控仪表盘（静态分析已足够）
- 1-1 以外的关卡

## 决策

### D1：SOP 结构——带决策门槛的分阶段设计
**选择：** 四个阶段（0→3），每个阶段有目标指标、固定的 episode 预算和通过/不通过门槛。
- 阶段 0：基线（100 eps，验证流水线）→ 门槛：avg_x_pos > 300
- 阶段 1：超参数扫描（6×100 eps，针对关键参数）→ 门槛：completion_rate ≥ 20%
- 阶段 2：算法调优（软更新、梯度裁剪、Dueling）→ 门槛：completion_rate ≥ 50%
- 阶段 3：最终打磨（200–500 eps，最佳配置）→ 目标：completion_rate ≥ 80%

**替代方案：** 一次性网格搜索所有超参数。已拒绝——组合过多（组合爆炸），浪费 GPU 时间。分阶段方法将算力集中在有前途的方向上。

### D2：分析工具采用独立脚本，而非 Jupyter Notebook
**选择：** `scripts/analyze.py` 和 `scripts/compare.py` 作为 CLI 工具，从实验目录读取 metrics.json/CSV。
**理由：** CLI 工具无需浏览器即可通过 SSH 使用，更容易与其他命令串联，产生适合终端查看的文本输出。Notebook 需要端口转发，增加复杂性。

### D3：指标聚合格式
**选择：** 继续使用每个实验各自的 `metrics.json` + `metrics.csv`。新增 `scripts/compare.py` 读取所有实验目录并生成统一对比表。
**理由：** 不引入新的数据库或格式。利用已有的 `MetricsLogger`。对比工具是薄聚合层。

### D4：通关优先的评估策略
**选择：** 当有评估数据时，基于 `eval_completion_rate` 选择检查点，回退到 `moving_avg_reward`。SOP 要求必须使用 `--eval-every` 参数（阶段 1–2 每 25 个 episode，阶段 3 每 10 个）。
**理由：** 奖励可以被刷（反复吃金币）。通关是二值的、无歧义的。

### D5：SOP 采用 Markdown 文档形式，而非自动化编排器
**选择：** `SOP.md` 是一份人可读的指南，包含阶段定义、超参数表和决策标准。`run_experiments.py` 为阶段 1 的扫描提供自动化。
**理由：** 阶段转换的完全自动化需要解读结果和做出判断——过早自动化。SOP.md 给研究者清晰的标准，而 `run_experiments.py` 省去手动输入命令的工作。

## 风险 / 权衡

- **[风险] 阶段门槛过于激进** → 研究者可能因过早放弃一个好的配置而浪费时间重新跑阶段。**缓解：** 门槛标准是软性建议；SOP.md 文档化何时可覆盖。
- **[风险] Dueling + 大 batch 导致 GPU 显存峰值** → 2080 Ti 有 22GB，Dueling（额外头）+ batch 128 仍在安全范围内。**缓解：** SOP 中限制 batch 为 64；首次运行时通过 `nvidia-smi` 监控验证。
- **[风险] `remote_train.py` 下载全部内容** → 权重文件每个约 10MB；每个实验后都下载会累积。**缓解：** 为 download 子命令添加 `--latest-only` 参数；尽可能在远端进行分析。
- **[权衡] 标准 DQN vs 先进方法** → DDQN + 软更新 + Dueling 可能不是最先进的，但已被证明对 Atari 级别问题稳定。在 1-1 上优先通关目标用 DDQN 是可以实现的。
