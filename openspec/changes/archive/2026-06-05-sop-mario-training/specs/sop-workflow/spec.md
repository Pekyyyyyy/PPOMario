## 新增需求

### 需求：SOP 定义分阶段训练流程
系统必须包含一份 `SOP.md` 文档，定义 4 阶段训练流程（阶段 0：基线、阶段 1：超参数扫描、阶段 2：算法调优、阶段 3：最终打磨）。每个阶段必须指定：episode 预算、超参数范围、评估频率、成功标准（门槛）以及推荐的 `train.py` 参数。

#### 场景：研究者首次阅读 SOP 开始训练
- **当** 研究者首次打开 `SOP.md`
- **则** 必须找到阶段 0 的清晰指令，包含具体运行命令、预期耗时，以及进入下一阶段的门槛标准（avg_x_pos > 300）

#### 场景：阶段门槛达成
- **当** 某个阶段的门槛标准已满足（例如阶段 1 的 completion_rate ≥ 20%）
- **则** SOP 必须指定下一阶段的超参数组合和命令

#### 场景：阶段门槛未达成
- **当** 在完整的 episode 预算后，某个阶段的门槛标准未被满足
- **则** SOP 必须提供回退指导：优先调整哪个超参数（gamma → learning rate → exploration decay）

### 需求：SOP 强制 RTX 2080 Ti 的资源约束
SOP 必须强制所有实验使用 batch_size ≤ 64、replay_buffer_size ≤ 200,000、stack_frames = 4，以避免 RTX 2080 Ti（22GB）上的 GPU 显存耗尽。

#### 场景：实验超出资源限制
- **当** 研究者尝试运行 batch_size > 64 的实验
- **则** SOP 必须警告这超出了 2080 Ti 的安全限制

### 需求：SOP 覆盖完整的实验生命周期
SOP 必须文档化完整工作流：(1) 检查 GPU 可用性，(2) 同步代码到远端，(3) 后台启动训练，(4) 通过 status 监控进度，(5) 下载结果，(6) 分析与对比。每一步必须引用具体的 CLI 命令。

#### 场景：研究者想监控正在运行的训练
- **当** 训练在远端服务器后台运行中
- **则** SOP 必须引导他们使用 `python remote_train.py status` 并解释如何解读输出

### 需求：SOP 定义评估优先的检查点选择策略
SOP 必须指定最佳检查点由 `eval_completion_rate`（来自周期性 `--eval-every` 运行）选择，而非训练奖励移动平均。若评估指标不可用，SOP 必须以 `moving_avg_reward` 作为第二选择标准。

#### 场景：两个检查点的移动平均奖励相近
- **当** 检查点 A 的 moving_avg_reward = 1800，检查点 B 的 moving_avg_reward = 1750
- **则** SOP 必须指示研究者用 10 轮贪婪 episode 评估两者，选择 completion_rate 更高的那个
